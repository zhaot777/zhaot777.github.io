Concurrency control

  1. Ensure serializability of transactions in a multiuser database environment
  • Problems
    • Lost update: update same data
    • Uncommitted data: one accessed uncommitted data, the other one rolled back
    • Inconsistent retrievals: transaction accesses on some data other is working on
  • Serializable schedule: Interleaved execution of transactions yields the same results as the serial execution of the transactions
  • Locking methods
    • Type
      • Binary Lock
      • Exclusive Lock(write)
      • Shared Lock(read)
    • Two-Phase Locking(2PL)
      • Defines how transactions acquire and relinquish locks, guarantees seralizability but does not prevent deadlocks
      • Growing phase - Transaction acquires all required locks without unlocking any data
      • Shrinking phase - Transaction releases all locks and cannot obtain any new lock
      • Rules
        • Two transactions cannot have conflicting locks
        • Cannot unlock before lock
        • No data are affected until all locks are obtained
  • Stamping methods
    • Assigns global, unique time stamp to each transaction
    • Properties: Uniqueness & Monotonicity
    • Disadvantages: increase memory, processing, resources, more fields
  • Optimistic methods
    • Transaction is validate to ensure that the changes made will not affect the integrity and consistency of the database
    • Use data resources without acquiring locks on those resources
