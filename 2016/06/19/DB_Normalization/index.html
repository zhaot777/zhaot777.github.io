Posted on 2016-06-19   |   In Database   |  

What is normalization and what role it plays in the DB design process

  1. Evaluating and correcting table structures to minimize data redundancies
  • Reduces data anomalies
  • Higher normal forms are better than lower normals forms and result in cleaner designs
  • Properly designed 3NF structures meet the requirement of 4NF
  • Normalization is a design step: Used to design a new DB structure or to improve the existing data structure
  • Normalization Goal
    • Each table represents a single subject
    • No data item will be unnecessarily stored in more than one table
    • All non-prime attribute in a table are dependent on the primary key
    • Each table is void of insertion, update and deletion anomalies

Normal Forms

  1. Characteristic
    • 1NF: Table format, no repeating groups, PK identified
    • 2NF: 1NF and no partial dependencies
    • 3NF: 2NF and no transitive dependencies
    • 4NF: 3NF and no independent multivalued dependencies
  • Process
    • Work on one relation at a time
    • Identifying the dependencies of a relation
    • Progressively breaking the relation into new set of relations
  • Dependencies
    • Functional Dependence: Each value of attribute A determines one and only one value of attribute B, then we say B is functionally dependent on A
    • Partial Dependency: functional dependence in which the determinant is only part of the PK
    • Transitive Dependency: An attribute functionally depends on another non-key attribute
  • Dependency diagram
    • Mark PK
    • Indicate full dependencies on the top and partial/transitive dependencies on the bottom
  • BCNF
    • Candidate key: same as primary key but not chosen to be the primary key
    • If the table contains more than one candidate key, it violated the BCNF, which is a special case of 3NF
    • Solution: try to change the primary key and separate one relation to two

Denormalization

  • Produces a lower normal form and increased performance and greater data redundancy
